## 사용자 전제 (필수)
- **사용자는 코딩 지식이 전혀 없음.** UI 기준 일반 용어로만 소통. 기술 명령어 안내 금지. 에이전트가 직접 실행.
- **승인 전 코드 수정 절대 금지 (AGENTS.md 섹션3 규칙0 준수).** 이 스킬이 호출되었다고 해서 자동으로 수정에 들어가는 것이 아님. 사용자가 "진행해/수정해/구현해/적용해/go" 등 명시적 실행 지시어를 준 경우에만 수정. 분석/조사/계획/추천까지만 수행하고 대기.

## 프론트엔드 수정 절차

### 1. 사전 조사 (필수 — AGENTS.md 섹션3 규칙 0-2 준수)
- 대상 파일의 전체 구조와 의존성 확인
- 영향 받는 모든 파일/함수/변수 식별
- 기존 코드 패턴과의 일관성 확인
- **아키텍처 원칙 점검 (AGENTS.md 섹션2 "코드 수정 시 점검 체크리스트 — 프론트엔드" 사용)**:
  - [ ] **P21 (사용자 투명성)**: 백엔드 상태 변화(매수 차단, 리스크 초과, 가드 조건 충족 등)를 UI에 표시했는가? 사용자가 "왜 이 종목이 매수되지 않았지?"라고 의문을 갖지 않도록 했는가?
  - [ ] **P23 (용어 통일)**: UI 텍스트가 용어 사전(`ARCHITECTURE.md` 부록 L)을 준수하는가? — "업종" not "섹터", "종목" not "주식", "매수 후보" not "바이 리스트", "보유 종목" not "홀딩/포지션"
  - [ ] **P23 (UI 패턴 일관성)**: 동일한 UI 패턴(목록/카드/태그/버튼/모달 등)이 2회 이상 반복 시 `components/common/` 하위 공통 컴포넌트로 추출했는가? 직접 중복 구현이 아닌가?
  - [ ] **P23 (공통 자산 재사용 — 사전)**: 신규 컴포넌트/색상/패턴/함수 구현 전, `components/common/` 및 기존 공통 함수/표준 색상을 먼저 검색했는가? 동일 또는 유사한 공통 자산이 있으면 반드시 그것을 활용하고 같은 기능을 새로 만들지 않았는가? (AGENTS.md 섹션3 규칙 0-2.4)
- **기존 로직 롤백 여부 확인 (AGENTS.md 섹션3 규칙 0-3/0-4/0-5 준수)**: 이 수정이 기존 UI 코드를 이전 상태로 되돌리는 행위(롤백)인지 확인. 롤백에 해당하면 사유·대상·영향 범위를 사용자에게 보고 후 승인받아야 함 (승인 없는 롤백 절대 금지). 핵심 화면(매수 후보 목록, 업종 점수 표시, 보유 종목 등) 표시 로직 변경 시 UI 기준 일반 용어로 변경 내용 설명 + 승인 필수(규칙 0-4). 사용자가 직접 설계/승인한 UI 패턴/표시 로직은 사유·영향·대안 상세 보고 후 승인 필수(규칙 0-5).
- 조사 결과(의존성/영향범위/원칙 부합)를 수정 계획에 포함하여 사용자에게 제시. 승인 후 수정 시작.

### 2. 코드 수정
- 작은 단위로 수정 (파일 하나씩, 블록 단위)
- 실시간 데이터 처리 금지 목록 준수:
  - innerHTML 전체 교체 금지 → 변경된 요소만 업데이트
  - .map() 전체 재생성 금지 → 변경된 항목만 교체
  - 전체 store 초기화 후 재설정 금지 → 해당 key만 업데이트
  - setInterval() 반복 전체 조회 금지 → WS 이벤트 수신으로 대체
  - setTimeout() 중첩 재귀 금지 → 단일 이벤트 리스너로 대체
  - alert(), confirm(), prompt() 금지
  - 대용량 JSON.stringify() 금지 → delta만 직렬화

### 3. 정적 검증
```bash
npm run type-check
npm run lint
```

### 4. 빌드 검증
```bash
npm run build
```
- 빌드 성공 확인
- 타입 오류 없음 확인

### 5. 브라우저 검증
- 개발 서버에서 실제 화면 확인
- 새로고침 후에도 정상 동작 확인
- 캐시/서비스워커 문제 없는지 확인

### 6. UI 문제 확인 순서
#### 버튼/입력값이 동작하지 않는 문제
1. 이벤트 핸들러가 연결되어 있는가
2. 핸들러가 호출되는가
3. store/state가 변경되는가
4. 변경된 state가 다시 UI에 반영되는가
5. 타입 오류나 값 변환 문제가 있는가

#### 화면이 예전 상태로 보이는 문제
1. 개발 서버가 최신 번들을 제공하는가
2. 브라우저 강력 새로고침을 했는가
3. 서비스워커가 등록되어 있는가
4. 캐시가 남아 있는가

### 7. 보고
- 수정한 파일
- 해결한 근본 원인
- 검증한 항목 (정적 검증 + 빌드 + 브라우저)
- 사용자가 직접 확인할 방법
- **단계 완료 시 작업 여력 보고 (AGENTS.md 섹션4 Context Management Rules 10 준수)**: 각 단계 완료 시 사용자에게 현재 작업 여력을 일반 용어로 보고 ("작업 여력 충분/적음"). 보고 후 커밋 + HANDOVER.md 갱신 진행 여부를 사용자 승인받아 진행.

## 주의사항
- 빌드 성공만으로 브라우저 표시가 100% 보장되는 것은 아님
- UI 문제는 실제 화면 확인 필요
- 실시간 수신 데이터는 delta만 처리, 50ms 초과 시 경고 로그, 200ms 초과 시 처리 중단 및 원인 보고
- **P21 (사용자 투명성)**: 백엔드 상태(매수 차단, 리스크 초과 등)를 UI에 표시하지 않으면 사용자가 원인을 알 수 없음 — 새 기능 추가 시 백엔드 상태 표시 여부 확인
- **P23 (용어 통일)**: UI 텍스트에 용어 사전(`ARCHITECTURE.md` 부록 L) 준수 — "업종" not "섹터", "종목" not "주식", "매수 후보" not "바이 리스트"

## 작업 중 발견 문제 기록 의무
- 메인 작업 도중 발견한 아키텍처 위반(P원칙), 오류, 잠재적 버그, dead code, 폴백 패턴, 아키텍처 원칙에 부합하는 더 나은 구조(개선점) 등은 즉시 `HANDOVER.md` "미해결 문제" 섹션에 기록 (파일:줄, 위반/부합 원칙 번호, 증상/개선내용). 사용자 승인 불필요 — 발견 즉시 기록. 상세 규칙은 AGENTS.md 섹션4 규칙 9 참조.
