## 사용자 전제 (필수)
- **사용자는 코딩 지식이 전혀 없음.** UI 기준 일반 용어로만 소통. 기술 명령어 안내 금지. 에이전트가 직접 실행.
- **승인 전 코드 수정 절대 금지 (AGENTS.md 섹션3 규칙0 준수).** 사용자가 "진행해/수정해/구현해/적용해/go" 등 명시적 실행 지시어를 준 경우에만 수정. 분석/조사/계획/추천까지만 수행하고 대기.

## 안전한 거래 코드 수정 절차

### 1. 수정 전 확인
> **다단계 작업 안내**: 거래 로직 신규 기능/구조 변경/다단계 작업인 경우 AGENTS.md 섹션4 "다단계 작업 워크플로우" 적용 (설계→태스크→구현 3세션). 본 스킬은 구현 단계(3세션~)의 거래 안전성 절차 담당.
- `TRADE_MODE`, `SIMULATION`, `TEST_MODE` 등의 설정값이 `test`/`simulation`/`true`인지 확인
- 브로커 API 키, 계좌 비밀번호, 토큰이 코드에 하드코딩되어 있지 않은지 `grep`으로 검색
- 실제 주문을 발생시키는 함수/엔드포인트가 `test_mode` 플래그로 보호되어 있는지 확인
- **기존 로직 롤백 여부 확인 (AGENTS.md 섹션3 규칙 0-3/0-4/0-5 준수 — 거래 로직은 최우선 엄격 적용)**: 이 수정이 기존 거래 로직(매수/매도 조건, 주문 경로, 리스크 검사, 업종 점수 계산 등)을 이전 상태로 되돌리는 행위(롤백)인지 확인. 롤백에 해당하면 사유·대상·영향 범위를 사용자에게 보고 후 승인받아야 함 (승인 없는 롤백 절대 금지). 거래 로직은 돈이 직결되므로 규칙 0-4(핵심 로직 변경 UI 기준 설명+승인)와 규칙 0-5(사용자 설계/승인 로직 더 엄격)를 다른 스킬보다 더 엄격하게 적용 — "dead code로 판단되어 제거" 등의 단독 사유로 사용자가 설계한 매매 조건/주문 경로를 승인 없이 되돌리는 것은 절대 금지.
- **사용자 의도 파악 질문 (모호함 감지 시 — problem-solve 스킬 섹션 1-1 기반 + 거래 특화 추가)**: 사전조사 완료 후 problem-solve 스킬 섹션 1-1의 기본 질문 카테고리(수치/기간 파라미터, UI 조작 위치/UX, 범위/예외 케이스, 우선순위/충돌, 검증 기준)를 먼저 적용한 후, 거래 로직 특화 질문 카테고리를 추가로 적용 — (1) 실전/모의 전환 여부 및 전환 시점, (2) 주문 경로(`execute_buy()`/`execute_sell()`)에 미치는 영향 범위, (3) 리스크 임계값(최대 매수 금액, 종목당 비중, 일일 손실 한도 등) 사용자 결정 값. 거래 로직은 돈이 직결되므로 모호성이 적어도 임계값/모드 전환 항목은 반드시 확인. 명확한 항목은 질문 생략 (과잉 질문 금지 — P24 단순성).

### 2. 아키텍처 원칙 준수 (필수 — 거래 로직은 돈이 직결되므로 최우선)

#### 2-1. 단일 주문 경로 (원칙 15)
- 주문은 `trading.py` → `execute_buy()` / `execute_sell()` 단일 경로만 사용
- **금지**: 주문 로직 분기 (예: 새로운 매수 함수 별도 생성, 라우터 우회 경로 추가)
- **금지**: 병렬/죽은 주문 경로 생성 (실행되지 않는 주문 코드 방치)
- 테스트모드: `dry_run.fake_send_order()` / 실전: `router.order.send_order()` — 이 2개만 허용

#### 2-2. 살아있는 경로 배선 (원칙 16)
- `RiskManager`는 `execute_buy()`/`execute_sell()` 내부에서 호출 — 외부에서 사전 체크만 하고 실제 주문 함수 내부 검사를 생략하는 것 금지
- `CircuitBreaker`는 주문 전후에 호출 — 주문 후 체크 생략 금지
- **금지**: 호출되지 않는 안전코드 작성 (예: RiskManager 체크 함수를 만들었으나 주문 경로에서 호출 안 함)
- **검증**: 수정 후 런타임 기동 시 RiskManager/CircuitBreaker 호출 로그 확인

#### 2-3. 테스트모드 동등성 (원칙 18)
- 테스트모드와 실전모드는 **돈 I/O만 다르고 그 외 모든 과정은 동일**
- **금지**: 테스트모드에서만 안전장치 생략 (예: "테스트니까 RiskManager 스킵")
- **금지**: 모드 분기를 돈 I/O 최소 지점(`dry_run.fake_send_order()`) 외에 추가
- 업종 점수 계산, 필터링, 타이밍 로직은 테스트모드/실전모드 동일하게 동작
- **검증**: 테스트모드에서 RiskManager/CircuitBreaker/필터링이 실제로 실행되는지 로그 확인

### 3. 허용되는 변경
- 모의투자/테스트용 매수/매도 로직 추가 (단, `execute_buy()`/`execute_sell()` 경로 내에서)
- 조건/임계값 수정
- 로그/알림 개선
- 주문 집계/필터링 로직 추가 (단, 단일 주문 경로 유지)

### 4. 금지되는 변경
- 실제 브로커 API 키 직접 입력
- `test_mode=False` 또는 `simulate=False` 등으로 강제 변경
- "모의투자"가 아닌 "실거래"로 설정 변경
- 사용자 승인 없이 주문 코드 실행
- 실제 돈이 들어가는 계좌로 테스트
- **주문 경로 분기 또는 우회 경로 생성 (원칙 15 위반)**
- **RiskManager/CircuitBreaker를 주문 함수 외부로 분리 (원칙 16 위반)**
- **테스트모드에서 안전장치 생략 (원칙 18 위반)**

### 5. 사용자가 실거래를 요청한 경우
- 반드시 경고: "실제 돈이 손실될 수 있습니다"
- 모의투자로 먼저 테스트할 것을 권유
- 사용자가 명확하게 "실제 계좌로 진행"이라고 승인한 경우에만 진행

### 6. 완료 보고
- 수정한 설정/변수
- 거래 모드 (test/simulation 여부)
- 실제 API 키 유무
- 사용자 확인 여부
- **원칙 15/16/18 준수 여부** — 주문 경로 단일성, 안전장치 배선, 테스트모드 동등성 각각 명시
- **용어 사전 준수 (P23)**: 사용자 보고, 로그 메시지 시 `ARCHITECTURE.md` 부록 L 표준 용어 사전 준수 — "매수" not "Buy/구매", "매도" not "Sell/판매", "매수 후보" not "바이 리스트"
- **단계 완료 시 작업 여력 보고 (AGENTS.md 섹션4 Context Management Rules 10 준수)**: 각 단계 완료 시 사용자에게 현재 작업 여력을 일반 용어로 보고 ("작업 여력 충분/적음"). 보고 후 커밋 + HANDOVER.md 갱신 진행 여부를 사용자 승인받아 진행.

⚠️ 사용자가 "실제 돈으로 테스트"라고 해도 한 번 더 경고하고 모의투자를 권장하라.

## 작업 중 발견 문제 기록 의무
- 메인 작업 도중 발견한 아키텍처 위반(P원칙), 오류, 잠재적 버그, dead code, 폴백 패턴, 아키텍처 원칙에 부합하는 더 나은 구조(개선점) 등은 즉시 `HANDOVER.md` "미해결 문제" 섹션에 기록 (파일:줄, 위반/부합 원칙 번호, 증상/개선내용). 사용자 승인 불필요 — 발견 즉시 기록. 상세 규칙은 AGENTS.md 섹션4 규칙 9 참조.
