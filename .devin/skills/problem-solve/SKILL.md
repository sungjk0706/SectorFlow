## 사용자 전제 (필수)
- **사용자는 코딩 지식이 전혀 없음.** UI 기준 일반 용어로만 소통. 기술 명령어(curl, npm, python, git 등) 안내 금지. 에이전트가 직접 실행.
- 사용자에게 확인/검증을 요청할 때는 항상 UI 조작으로 안내. 터미널/콘솔 확인 요청 금지.

## 문제 해결 절차

### 1. 사전 조사 (필수)
- 대상 파일의 전체 구조와 의존성 확인
- 영향 받는 모든 파일/함수/변수 식별
- 기존 코드 패턴과의 일관성 확인
- 테스트 커버리지 및 기존 테스트 영향 확인
- 아키텍처 원칙 (SSOT, 살아있는 경로, 폴백 금지) 부합 여부 확인

### 2. 요청 유형 분류 (승인 전 수정 절대 금지)
- **보고만 원함**: 조사 후 원인과 해결책 보고, 코드 수정 금지
- **추천/의견 요청**: "추천해줘", "너의 추천은?", "어떻게 생각해", "의견 줘" 등 — 추천/의견 제시 후 대기. 추천을 제시했다고 해서 자동으로 수정에 들어가는 것은 절대 금지. 사용자가 명시적으로 "진행해/수정해/적용해" 등 실행 지시어를 줄 때까지 대기
- **수정까지 원함**: 사용자가 "진행해/수정해/구현해/적용해/go" 등 명시적 실행 지시어를 사용한 경우에만 수정

### 3. 재현 범위 확인
- 특정 페이지만 문제인가?
- 전체 앱인가?
- 개발 서버 vs 빌드 결과인가?
- 새로고침 후에도 반복되는가?

### 4. 진입점 확인
- 프론트엔드: `frontend/src/main.ts`, `frontend/src/layout/shell.ts`
- route 등록, container 렌더링, props/state 전달, CSS 숨김 여부 확인

### 5. 데이터 흐름 확인
- API 응답 → Store → Component → DOM
- WebSocket → binding → Store → Component → DOM
- 각 단계에서 값이 올바르게 흐르는지 추적

### 6. 원인 후보 분리
- 확인된 실패 지점을 원인으로 좁힘
- 최소 수정으로 해당 경로만 복구
- 주변 코드에 영향을 주지 않도록 주의

### 7. 근본 해결
- 증상을 숨기는 임시방편 금지
- 금지: z-index 무작정 올림, `!important`, `height` 임의 추가, `as any`로 덮기, 새로고침만 안내, `service worker` 캐시 문제 무시하고 코드만 수정, 사용자 승인 없이 범위 밖 리팩토링, UI 디자인 임의 변경
- 허용: props/state 전달 경로, CSS 충돌, flex 스크롤 체인, 캐시/서비스워커 여부 확인

### 7-1. 작은 단위 진행 원칙 (상세)
- 파일 하나씩 수정, 큰 파일은 블록 단위로 수정
- 수정 후 바로 검색 검증, 중간 상태를 짧게 보고
- **금지**: 긴 python 치환 스크립트로 여러 파일 한 번에 변경, 큰 apply_patch로 반복 문자열 무리하게 치환, 정규식 하나에 많은 조건을 섞어 검색
- **허용**: 파일 읽기, 작은 패치 적용, 고정 문자열 검색으로 잔여 확인, 다음 파일 진행, 빌드 검증

### 7-2. 실패를 줄이는 검색 원칙
복잡한 정규식은 오류를 만들 수 있음 (특히 `{ } ( ) [ ] . * + ? | \`).
- 정확한 문자열은 고정 문자열 검색 사용
- 여러 조건을 한 번에 검색하지 말고 나눠서 검색
- 검색 결과가 많으면 범위를 좁힘

### 8. 검증
- 잔여 문자열 검색
- 타입 체크, 빌드, 테스트
- 백엔드 수정 시 런타임 기동 검증 필수
- 프론트엔드 수정 시 `npm run build` + 브라우저 화면 확인

### 8-1. 출력 최적화 (토큰 절약)
- 추론/생각 과정 출력 금지. "제 생각에는", "여기서 고민해볼 점은" 등 표현 사용 금지.
- 최종 검증된 사실과 결론만 보고.
- 조사 결과는 간결하게 요약: 호출 경로는 A→B→C 체인으로, 영향 파일은 파일명만 나열, 근본 원인은 파일:줄번호 + 한 문장, 수정안은 변경 전/후 스니펫만.

### 9. 보고 (필수 포함)
1. 문제 현상
2. 근본 원인 (파일:줄)
3. 수정 방안
4. 수정 영향 범위
5. 검증 방법
6. 추가로 발견한 문제/개선점 — 메인 작업과 무관하게 도중 발견한 아키텍처 위반(P원칙), 오류, 잠재적 버그, dead code, 폴백 패턴 등. 발견 즉시 `HANDOVER.md` "미해결 문제" 섹션에 기록 (파일:줄, 위반 원칙 번호, 증상). 사용자는 코딩 지식이 없어 직접 해결 불가하므로 기록 누락 = 영영 미해결.

### 10. 후속 작업
- 사용자 승인 후에만 Git 커밋
- 사용자 승인 후에만 `HANDOVER.md` 업데이트
- 단, "작업 중 발견 문제 기록 의무"(AGENTS.md 섹션4 규칙 9)는 예외 — 발견 즉시 기록, 사용자 승인 불필요

### 11. 세션당 1단계 원칙
- 각 세션은 1단계(Step)만 진행
- 단계 완료 시: 커밋 + HANDOVER.md 업데이트 + 사용자 보고
- 작업량이 많은 경우 사전 분할 후 사용자에게 분할 계획 보고
