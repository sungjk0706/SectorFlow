## 사용자 전제 (필수)
- **사용자는 코딩 지식이 전혀 없음.** UI 기준 일반 용어로만 소통. 기술 명령어(curl, npm, python, git 등) 안내 금지. 에이전트가 직접 실행.
- 사용자에게 확인/검증을 요청할 때는 항상 UI 조작으로 안내. 터미널/콘솔 확인 요청 금지.

## 문제 해결 절차

### 1. 사전 조사 (필수 — AGENTS.md 섹션3 규칙 0-2 준수)
> **다단계 작업 안내**: 신규 기능/구조 변경/다단계 작업인 경우 AGENTS.md 섹션4 "다단계 작업 워크플로우" 적용 (설계→태스크→구현 3세션). 본 스킬은 구현 단계(3세션~)의 사전조사·근본 해결 절차 담당.
- 대상 파일의 전체 구조와 의존성 확인
- 영향 받는 모든 파일/함수/변수 식별
- 기존 코드 패턴과의 일관성 확인
- 테스트 커버리지 및 기존 테스트 영향 확인
- **아키텍처 원칙 점검 (구체적 — AGENTS.md 섹션2 "코드 수정 시 점검 체크리스트" 사용)**:
  - [ ] **P10 (SSOT)**: 데이터가 한 곳에서만 관리되는가? 중복 저장/독립 관리가 없는가?
  - [ ] **P16 (살아있는 경로)**: 안전장치가 실제 실행 경로에 연결되었는가? 호출되지 않는 dead code가 아닌가?
  - [ ] **P20 (폴백 금지)**: 정상 경로의 빈 문자열/None/누락을 폴백으로 덮는 분기가 있는가? silent `except: pass`가 있는가?
  - [ ] **P21 (사용자 투명성)**: 사용자가 모르는 중요 의사결정이 있는가? 백엔드 상태 변화를 UI에 표시해야 하는가?
  - [ ] **P22 (데이터 정합성)**: 파생 데이터가 중복 저장되지 않는가? 원본에서 파생하는가? 불일치 시 즉시 차단하는가?
  - [ ] **P23 (일관성)**: 같은 기능이 파일 간 다른 패턴으로 구현되지 않았는가? 용어 사전(부록 L)을 준수하는가?
  - [ ] **P23 (공통 자산 재사용 — 사전)**: 신규 함수/컴포넌트/상수/색상/패턴 구현 전, 기존 공통 자산(백엔드: `core/constants.py`·공통 유틸·기존 함수 / 프론트엔드: `components/common/`·공통 함수·표준 색상)을 먼저 검색했는가? 동일 또는 유사한 공통 자산이 있으면 반드시 그것을 활용하고 같은 기능을 새로 만들지 않았는가? (AGENTS.md 섹션3 규칙 0-2.4)
  - [ ] **P24 (단순성)**: 같은 기능을 더 단순한 방법으로 구현할 수 있는가? 함수 50줄/파일 500줄/복잡도 10 이하?
- **기존 로직 롤백 여부 확인 (AGENTS.md 섹션3 규칙 0-3/0-4/0-5 준수)**: 이 수정이 기존 코드를 이전 상태로 되돌리는 행위(롤백)인지 확인. 롤백에 해당하면 사유·대상·영향 범위를 사용자에게 보고 후 승인받아야 함 (승인 없는 롤백 절대 금지). 수신률·업종 점수·매매 로직·매수 후보 선정·매도 조건 등 핵심 기능 변경 시 UI 기준 일반 용어로 변경 내용 설명 + 승인 필수(규칙 0-4). 사용자가 직접 설계/승인한 로직(예: 누적 수신 세트, 업종 점수 계산식)은 사유·영향·대안 상세 보고 후 승인 필수(규칙 0-5).
- 조사 결과(의존성/영향범위/원칙 부합)를 수정 계획에 포함하여 사용자에게 제시. 승인 후 수정 시작.

### 1-1. 사용자 의도 파악 질문 프로세스 (모호함 감지 시 — AGENTS.md 섹션3 규칙 0-2 보완)

사전조사(섹션1) 완료 후, 사용자 지시에서 모호한 항목이 있으면 아래 카테고리에서 선별하여 질문. **명확한 항목은 질문 생략** (과잉 질문 금지 — P24 단순성).

#### 트리거 (질문 수행 조건)
- 사용자 지시에 수치/기간/위치/범위/예외처리 중 하나라도 명확하지 않은 경우
- 기존 코드 패턴으로 자동 결정할 수 없는 비즈니스 의도가 포함된 경우
- 단일 버그 수정이더라도 수정 범위에 위 항목이 걸치면 적용 (경량 적용 — 모호한 것만 1~2개 질문)

#### 질문 카테고리 (우선순위 순 — 위에서부터 먼저 검토)
1. **수치/기간 파라미터**: 사용자만 정할 수 있는 값 (가산점 유지 시간, 임계값, 노출 개수, 대기 시간 등). 기존 상수/설정에 없으면 필수 질문.
2. **UI 조작 위치/UX**: 사용자가 어디서 조작/확인할지 (편집 화면, 표시 위치, 버튼 배치 등). 새 화면/새 진입점이면 필수 질문.
3. **범위/예외 케이스**: 정상 경로 외 처리 (대상 외 종목, 빈 데이터, 조건 불충족 시 동작). P20(폴백 금지)과 직결 — 묻지 않으면 폴백으로 덮게 됨.
4. **우선순위/충돌**: 기존 기능과 충돌 시 어느 쪽이 우선인지. 충돌이 감지된 경우만 질문.
5. **검증 기준**: 완료 후 사용자가 UI에서 어떻게 확인할지. 새 기능이거나 확인 방법이 자명하지 않은 경우만 질문.

#### 질문 작성 원칙
- **UI 기준 일반 용어 사용** (AGENTS.md 섹션3 사용자 의사소통 규칙 4 준수) — 기술 식별자, 함수명, 파일 경로, FID 번호 노출 금지.
- **선택지 + 트레이드오프 함께 제시** — 단답형이 아닌 "A로 하면 X, B로 하면 Y" 형태. 사용자가 코딩 지식이 없으므로 선택지 없는 개방형 질문은 답변 불가.
- **한 번에 1~4개 묶어서 질문** (ask_user_question 도구 활용). 5개 카테고리를 한 번에 모두 묻는 것 금지 — 모호한 것만 선별.
- **이미 코드/설정에 있는 값은 묻지 않음** — 사전조사에서 기존 상수/설정/패턴으로 정해지는 것은 자체 결정 (아래 "자체 결정 기준" 참조).

#### 자체 결정 기준 (질문하지 않고 에이전트가 정하는 것 — P24 단순성)
- 기존 공통 자산/패턴(`core/constants.py`, `components/common/` 등)으로 정해지는 구현 방식
- 아키텍처 원칙(P1~P25)이 강제하는 구조
- 사용자가 알 필요 없는 내부 처리 순서/네이밍/파일 배치
- 기존 코드와 동일한 패턴으로 자연스럽게 따르는 부분

#### 적용 범위
- **다단계/신규 기능**: 5개 카테고리 전부 검토, 모호한 것 선별 질문 (다단계 워크플로우 1세션 설계 단계에서 주로 적용)
- **단일 버그 수정/1파일 수정**: 모호성 감지 시에만 1~2개 질문 (경량 적용). 명확하면 질문 0건도 허용.
- **거래 로직 수정**: 본 프로세스 적용 후 safe-trade의 거래 특화 질문 카테고리(실전/모의 전환, 주문 경로 영향, 리스크 임계값)를 추가로 적용 — safe-trade 스킬 참조.

#### 질문 결과 기록
- 다단계 작업인 경우: 질문과 답변을 디자인 파일(`docs/architecture_*_design.md`)의 "사용자 결정 항목" 섹션에 기록. 2세션(태스크 파일)에서 그대로 활용.
- 단일 세션인 경우: 수정 계획 보고서에 "사용자 결정 항목"으로 포함하여 제시.

### 2. 요청 유형 분류 (승인 전 수정 절대 금지)
- **보고만 원함**: 조사 후 원인과 해결책 보고, 코드 수정 금지
- **추천/의견 요청**: "추천해줘", "너의 추천은?", "어떻게 생각해", "의견 줘" 등 — 추천/의견 제시 후 대기. 추천을 제시했다고 해서 자동으로 수정에 들어가는 것은 절대 금지. 사용자가 명시적으로 "진행해/수정해/적용해" 등 실행 지시어를 줄 때까지 대기
- **수정까지 원함**: 사용자가 "진행해/수정해/구현해/적용해/go" 등 명시적 실행 지시어를 사용한 경우에만 수정

### 3. 재현 범위 확인
- 특정 페이지만 문제인가?
- 전체 앱인가?
- 개발 서버 vs 빌드 결과인가?
- 새로고침 후에도 반복되는가?

### 4. 진입점 확인
- 프론트엔드: `frontend/src/main.ts`, `frontend/src/layout/shell.ts`
- route 등록, container 렌더링, props/state 전달, CSS 숨김 여부 확인

### 5. 데이터 흐름 확인
- API 응답 → Store → Component → DOM
- WebSocket → binding → Store → Component → DOM
- 각 단계에서 값이 올바르게 흐르는지 추적

### 6. 원인 후보 분리
- 확인된 실패 지점을 원인으로 좁힘
- 최소 수정으로 해당 경로만 복구
- 주변 코드에 영향을 주지 않도록 주의

### 7. 근본 해결
- 증상을 숨기는 임시방편 금지
- 금지: z-index 무작정 올림, `!important`, `height` 임의 추가, `as any`로 덮기, 새로고침만 안내, `service worker` 캐시 문제 무시하고 코드만 수정, 사용자 승인 없이 범위 밖 리팩토링, UI 디자인 임의 변경
- 허용: props/state 전달 경로, CSS 충돌, flex 스크롤 체인, 캐시/서비스워커 여부 확인
- **롤백으로 증상 덮기 금지**: 롤백 후에도 근본 원인이 남아있으면 재발. 롤백이 적절한 경우(잘못된 변경 되돌림 — 사유·대상·영향 보고 후 승인받은 경우, AGENTS.md 섹션3 규칙 0-3)와 부적절한 경우(증상 회피용 롤백)를 구분. 증상을 없애기 위해 이전 상태로 되돌리는 것은 근본 해결이 아님 — 원인을 식별하고 수정.

### 7-1. 작은 단위 진행 원칙 (상세)
- 파일 하나씩 수정, 큰 파일은 블록 단위로 수정
- 수정 후 바로 검색 검증, 중간 상태를 짧게 보고
- **금지**: 긴 python 치환 스크립트로 여러 파일 한 번에 변경, 큰 apply_patch로 반복 문자열 무리하게 치환, 정규식 하나에 많은 조건을 섞어 검색
- **허용**: 파일 읽기, 작은 패치 적용, 고정 문자열 검색으로 잔여 확인, 다음 파일 진행, 빌드 검증

### 7-2. 실패를 줄이는 검색 원칙
복잡한 정규식은 오류를 만들 수 있음 (특히 `{ } ( ) [ ] . * + ? | \`).
- 정확한 문자열은 고정 문자열 검색 사용
- 여러 조건을 한 번에 검색하지 말고 나눠서 검색
- 검색 결과가 많으면 범위를 좁힘

### 8. 검증
- 잔여 문자열 검색
- 타입 체크, 빌드, 테스트
- 백엔드 수정 시 런타임 기동 검증 필수
- 프론트엔드 수정 시 `npm run build` + 브라우저 화면 확인

### 8-1. 출력 최적화 (토큰 절약)
- 추론/생각 과정 출력 금지. "제 생각에는", "여기서 고민해볼 점은" 등 표현 사용 금지.
- 최종 검증된 사실과 결론만 보고.
- 조사 결과는 간결하게 요약: 호출 경로는 A→B→C 체인으로, 영향 파일은 파일명만 나열, 근본 원인은 파일:줄번호 + 한 문장, 수정안은 변경 전/후 스니펫만.

### 9. 보고 (필수 포함)

2. 근본 원인 (파일:줄)
3. 수정 방안
4. 수정 영향 범위
5. 검증 방법
6. 추가로 발견한 문제/개선점 — 메인 작업과 무관하게 도중 발견한 아키텍처 위반(P원칙), 오류, 잠재적 버그, dead code, 폴백 패턴, 아키텍처 원칙에 부합하는 더 나은 구조(개선점) 등. 발견 즉시 `HANDOVER.md` "미해결 문제" 섹션에 기록 (파일:줄, 위반/부합 원칙 번호, 증상/개선내용). 사용자는 코딩 지식이 없어 직접 해결 불가하므로 기록 누락 = 영영 미해결. 개선점은 객관적 근거 있는 것만 (P24 준수, 상세는 AGENTS.md 섹션4 규칙 9 "개선점 인정 기준" 참조).
7. **용어 사전 준수 (P23)**: 사용자 보고, 로그 메시지, 문서 작성 시 `ARCHITECTURE.md` 부록 L 표준 용어 사전 준수 — "업종" not "섹터", "종목" not "주식", "매수 후보" not "바이 리스트", "보유 종목" not "홀딩/포지션".

### 10. 후속 작업

### 11. 세션당 1단계 원칙 (AGENTS.md 섹션3 규칙 0-1 준수)
- 각 세션은 1단계(Step)만 진행. 한 세션에서 여러 단계 연속 진행 금지.
- 단계 완료 후 반드시 검증(테스트 + 런타임 기동/빌드) 수행. 이상 없으면 커밋 + HANDOVER.md 업데이트 + 사용자 보고 후 세션 종료.
- 다음 단계는 다음 세션에서 HANDOVER.md 기반으로 이어서 진행.
- 작업량이 많은 경우 사전 분할 후 사용자에게 분할 계획 보고.

### 11-1. 단계 완료 시 작업 여력 보고 (AGENTS.md 섹션4 Context Management Rules 10 준수)
- 각 단계 완료 시 사용자에게 현재 작업 여력을 일반 용어로 보고 ("작업 여력 충분/적음").
- 보고 후 커밋 + HANDOVER.md 갱신 진행 여부를 사용자 승인받아 진행.
- 세션 종료 시 보고(12항)와 구분 — 본 항목은 매 단계 완료 시점, 12항은 세션 종료 시점.

### 12. 세션 종료 전 잔존 프로세스 정리 (AGENTS.md 섹션3 규칙 5-1 준수)
- 세션 종료 전 반드시 백엔드/테스트 잔존 프로세스 0건 확인. 프로세스를 띄운 작업(런타임 기동, pytest, db-backup 등)이 없어도 확인 후 종료.
- 상세 절차는 AGENTS.md 섹션3 규칙 5-1 참조.
