## 아키텍처 원칙 (1~20)

### 아키텍처 원칙 (1~14)
1. **단일 asyncio 이벤트 루프**: 모든 비동기 작업은 단일 이벤트 루프 내에서 처리
2. **모든 I/O는 async def**: 동기 함수 금지, httpx.AsyncClient/aiosqlite 사용 (동기 requests/blocking 금지)
3. **run_in_executor 우회 금지**: 동기 코드를 비동기로 위장하는 행위 금지
4. **증권사 이름 공통 기능 침투 금지**: BrokerInterface/레지스트리 추상화로 격리
5. **EventBus/발행구독 패턴 사용 금지**: 직접 호출 체인 유지, asyncio.Queue 파이프라인으로 일원화 (콜백 리스트 옵서버 우회도 위반)
6. **SQLite 단일화**: ORM·무거운 추상화 금지, Raw SQL 직통 사용
7. **블로킹 = 지연 = 왜곡 = 망함**: 블로킹은 지연 유발, 데이터 왜곡, 시스템 붕괴
8. **실시간 파이프라인과 배치 파이프라인 분리**: 두 파이프라인 명확히 분리
9. **각 파이프라인 독립**: 파이프라인 간 상호 의존/간섭 금지
10. **단일 소스 진리(SSOT)**: 같은 데이터는 한 곳에서만 관리
11. **이벤트 기반 루프**: 폴링 금지, 이벤트 기반으로만 처리
12. **DB 연결 매번 생성/파괴 금지**: 앱 시작 시 1회 생성, 앱 종료 시까지 유지
13. **설정 매번 DB 쿼리 금지**: 설정은 메모리 상주
14. **멀티스레드 남용 금지**: 단일 루프 사용

### 실전 안전성 원칙 (15~19)
15. **단일 주문 경로**: 주문 경로는 하나만. 병렬/죽은 주문 경로 금지
16. **구현 = 살아있는 경로에 배선됨**: 안전/제어 장치는 실제 실행 경로에 연결되어 동작 입증돼야 함. 호출 안 되는 안전코드는 위험한 착시
17. **플래그 단일 소스**: 한 플래그는 정의·설정·읽기가 같은 변수를 가리켜야 함
18. **테스트모드 동등성**: 테스트모드와 실전모드는 돈 I/O만 다르고 나머지 모든 과정 동일. 모드 분기는 돈 I/O 최소 지점에만
19. **런타임 검증 게이트**: py_compile/import 성공은 검증이 아님. 실제 실행경로를 흘려보는 런타임 점검 + RuntimeWarning(coroutine never awaited)을 error로 승격한 테스트로 검증

### 근본 해결 원칙 (20)
20. **폴백 금지**: 폴백 코드는 정상 경로의 불안정성을 은폐하고 근본 원인 해결을 방해함
    - 정상 경로에서 절대 발생하지 않아야 할 상태(빈 문자열, None, 누락)를 폴백으로 덮지 말 것
    - 폴백 분기가 필요하다는 것 자체가 상위 초기화나 데이터 흐름에 결함이 있다는 신호
    - 결함이 발견되면 폴백으로 덮지 말고 원인을 제거할 것
    - 부득이한 예외 처리는 silent fallback이 아니라 에러 로그 출력하여 즉시 인지 가능하게 할 것
    - SSOT 원칙(원칙 10)과 충돌: 폴백은 사실상 제2의 데이터 소스를 만드는 것

## 핵심 구현 원칙
- **DB 연결 생명주기 공유**: 앱과 생명주기 공유, 매번 생성/파괴 금지
- **사용자 설정 즉시 반영**: 설정 변경 즉시 반영, 실시간 데이터 흐름 차단 금지
- **SQLite WAL 모드**: `PRAGMA journal_mode=WAL`, `synchronous=NORMAL`, `cache_size=-64000`, `temp_store=MEMORY` — 동시 읽기/쓰기 보장, 지연 스파이크 방어
- **DB 쓰기 직렬화**: 전용 db_write_queue + 단일 Background DB Writer 태스크로 순차 일괄 쓰기
- **GC 런타임 회피**: 장 시작 전 `gc.disable()`, 장중 유휴 시간/마감 후 `gc.collect()` 수동 호출
- **메모리 누수 방지**: 슬라이딩 윈도우/ring buffer 구조, tracemalloc 주기적 샘플링
