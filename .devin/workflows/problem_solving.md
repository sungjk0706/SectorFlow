## 진행 원칙 (필수 준수)

### 1. 사전조사 선행 원칙
Step 진행 전 반드시 다음 사항을 코드 기반으로 조사한다:
- 대상 파일의 전체 구조와 의존성
- 영향 받는 모든 파일/함수/변수
- 기존 코드 패턴과의 일관성
- 테스트 커버리지 및 기존 테스트 영향
- 아키텍처 원칙(SSOT, 살아있는 경로, 폴백 금지) 부합 여부

### 2. 세션당 1단계 원칙
- 각 세션은 1단계(Step)만 진행
- 단계 완료 시: 커밋 + HANDOVER.md 업데이트 + 사용자 보고

## 에이전트 강제 조항

### 조항 1: 추측/짐작 완전 배제, 코드 기반 사실 조사 의무화
에이전트는 로직 구현, 디버깅, 수정안 제시 시 추측이나 짐작을 완전히 배제한다. 모든 판단은 실제 코드베이스의 의존성, 영향 범위, 데이터 흐름을 광범위하게 조사한 사실 기반으로만 보고해야 한다.

구체적 행동 지침:
- 전체 호출 그래프 추적 (어디서 호출되고, 어디로 전파되는지)
- 영향 받는 모든 파일/함수/변수 식별
- 기존 코드 패턴과의 일관성 확인
- 테스트 커버리지 및 기존 테스트 영향 검토
- "아마", "아마도", "대부분", "일반적으로" 같은 추측어 사용 금지
- "이렇게 하면 될 것 같다" 식의 비검증 제안 금지

### 조항 2: 근본 해결책만 제시, 아키텍처 부합 의무화
에이전트가 수정 방안을 제시할 때는 반드시 아키텍처 원칙(20개)에 부합하면서 오류의 근본 원인을 완벽하게 해결하는 해결책만을 사용자에게 제시해야 한다. 임시방편, 우회책, 폴백은 금지한다.

구체적 행동 지침:
- 원칙 10 (SSOT): 단일 진실 원천 준수 여부 확인
- 원칙 16 (살아있는 경로): 실제 실행되는 코드 수정 여부 확인
- 원칙 20 (폴백 금지): 임시방편 아닌 근본 해결 여부 확인
- 수정 후 전체 영향 범위 재검증
- "임시로", "일단", "우선" 같은 임시방편 제안 금지
- "추후 개선", "나중에 리팩토링" 같은 미루기 금지

### 조항 3: 보고 형식 표준화 (필수 포함 항목)
에이전트는 다음 5개 항목을 항상 포함하는 형식으로 보고해야 한다.

필수 포함 항목:
1. 문제 현상 (UI 기준 설명 + 비유 + 실제 화면)
2. 근본 원인 (코드 기반, 파일:줄, 구체적 데이터 흐름)
3. 수정 방안 (아키텍처 원칙 부합 근거 포함)
4. 수정 영향 범위 (변경 파일, 영향 받는 다른 모듈)
5. 검증 방법 (테스트, 사용자 확인 방법)

### 조항 4: 출력 최적화 (토큰 절약) 및 후속 작업 의무화

4-1. 추론/생각 과정 출력 금지
- 자신의 추론, 가정, 생각 과정을 출력하지 않는다.
- "제 생각에는", "아마도", "여기서 고민해볼 점은" 등의 표현을 사용하지 않는다.
- 최종 검증된 사실과 결론만 보고한다.

4-2. 조사 결과는 간결하게 요약
- 호출 경로: 다이어그램 또는 간결한 텍스트 체인으로 표현 (A→B→C)
- 영향 파일: 파일명만 나열 (전체 코드 미출력)
- 근본 원인: 파일:줄번호 + 한 문장 설명
- 수정안: 변경 전/후 코드 스니펫만 (전체 파일 미출력)
- 중복되는 내용은 생략하고 핵심만 반복 없이 보고한다.

4-3. 근본 해결 디버깅 완료 후 의무 후속 작업
에이전트는 수정 작업이 완료되고 모든 테스트가 통과된 후 다음 2가지 작업을 반드시 수행한다:

(1) Git 커밋 및 푸시
- 수정된 파일들을 의미 있는 단위로 커밋한다.
- 커밋 메시지 형식: [유형] 간결한 설명 (50자 이내) + 본문에 파일 목록/근본 원인/검증 결과 포함

(2) HANDOVER.md 업데이트
- "최근 변경 사항" 섹션에 날짜, 수정자, 변경 내용, 영향 범위 기록
- "알려진 이슈" 섹션에서 해결된 이슈 제거 또는 "해결됨" 표시
- 필요한 경우 "설정/환경" 섹션 업데이트

※ 두 작업 모두 사용자 승인을 받은 후 실행한다. 승인 전까지는 커밋/푸시 및 HANDOVER 업데이트를 수행하지 않는다.

## 조사/분석 표준 순서

### 핵심 흐름
현상 확인 → 진입점 확인 → 데이터 흐름 확인 → 원인 후보 분리 → 최소 수정 → 검증

### 1단계: 사용자의 요청 유형 분류
**보고만 원하는 경우:**
- 파일 조사 가능
- 명령 실행은 읽기/검증 중심
- 코드 수정 금지
- 원인과 해결책 보고 후 대기

**수정까지 원하는 경우:**
- 조사 후 수정 가능
- 작은 단위로 진행
- 수정 후 검증 필수

### 2단계: 재현 범위 확인
- 특정 페이지만 문제인가
- 전체 앱 문제인가
- 개발 서버에서만 문제인가
- 빌드 결과에서도 문제인가
- 새로고침 후에도 문제인가
- 특정 브라우저 캐시 문제인가

### 3단계: 진입점 확인
**프론트엔드:**
- frontend/src/main.ts
- frontend/src/layout/shell.ts

**확인 항목:**
- route가 등록되어 있는가
- 현재 URL hash와 route가 일치하는가
- container component가 렌더링되는가
- shell.ts가 children을 표시하는가
- props/state가 전달되는가
- 스크롤 체인이 끊겼는가
- CSS가 숨기고 있는가

### 4단계: 데이터 흐름 확인
- API 응답 → Store → Component → DOM
- WebSocket 이벤트 → binding.ts → Store → Component → DOM
- 확인: API/WS 데이터 올바른가, Store 저장되는가, Store 변경이 컴포넌트에 반영되는가, DOM에 렌더링되는가

### 5단계: 원인 후보 분리
- 확인된 실패 지점을 원인으로 좁힘
- 최소 수정으로 해당 경로만 복구
- 주변 코드에 영향 주지 않음

## 근본해결 원칙

### 근본해결 정의
증상을 잠시 숨기는 것이 아니라, 같은 문제가 다시 발생하지 않도록 원인 경로를 제거하는 것

**금지:**
- z-index 무작정 올림
- !important 추가
- height 임의 추가
- 새로고침만 안내
- 임시 display 변경

**허용:**
- 코드상 렌더링 여부 확인
- props/state 전달 경로 확인
- CSS 충돌 경로 확인
- flex 스크롤 체인 확인
- 캐시/서비스워커 여부 확인
- 원인별 충돌 경로 제거

### 코드 기반 조사 원칙
추측으로 결론 금지, 반드시 다음을 근거로 삼음:
- 실제 파일 내용
- 검색 결과
- 빌드 로그
- 런타임 로그
- 브라우저에서 재현 가능한 현상
- store/state/data flow 확인 결과

**금지 표현:** "아마도", "그럴 것 같습니다", "일단 해보죠"
**허용 표현:** "파일 A의 n번째 줄에서 확인됨", "검색 결과 해당 클래스가 남아 있음", "빌드 로그에서 타입 오류가 확인됨", "이 부분은 아직 확인 필요"

### 보고와 수정의 분리
사용자가 "원인 분석 후 보고하라"고 하면 수정 금지

**올바른 순서:**
1. 코드 조사
2. 원인 보고
3. 해결 방안 제시
4. 사용자 승인 확인
5. 수정
6. 검증
7. 결과 보고

사용자가 명시적으로 "진행해", "수정해", "근본해결책으로 진행해"라고 승인한 경우에만 수정

### 작은 단위 진행 원칙
- 파일 하나씩 수정
- 큰 파일은 블록 단위로 수정
- 수정 후 바로 검색 검증
- 중간 상태를 짧게 보고
- 오래 걸릴 수 있는 명령은 사전에 알림

**금지:**
- 긴 python 치환 스크립트로 여러 파일 한 번에 변경
- 큰 apply_patch로 반복 문자열 무리하게 치환
- 정규식 하나에 많은 조건을 섞어 검색

**허용:**
- 파일 읽기
- 작은 패치 적용
- 고정 문자열 검색으로 잔여 확인
- 다음 파일 진행
- 빌드 검증

### 실패를 줄이는 검색 원칙
복잡한 정규식은 오류를 만들 수 있음 (특히 { } ( ) [ ] . * + ? | \)

**안전한 검색 방식:**
- 정확한 문자열은 고정 문자열 검색 사용
- 여러 조건을 한 번에 검색하지 말고 나눠서 검색
- 검색 결과가 많으면 범위를 좁힘

### 검증 없는 완료 보고 금지
수정 후 다음 중 최소 하나 이상 확인:
- 잔여 문자열 검색
- 타입 체크
- 빌드
- 테스트
- 브라우저 재현 확인
- **런타임 기동 확인 (백엔드 수정 시 필수)**

**런타임 기동 검증 (백엔드 수정 시 필수, 원칙 19 준수):**
- py_compile 통과와 pytest 통과는 런타임 동작을 보장하지 않음
- 백엔드 코드 수정 시 반드시 앱 기동 후 런타임 로그 확인
- 확인 항목:
  1. `.venv/bin/python main.py` 기동 — 정상 시작 여부
  2. 콘솔 로그 — 에러/Traceback/RuntimeWarning 없음
  3. 파일 로그 (`backend/logs/trading_*.log`) — 정상 기록 여부
  4. 기동 후 10~30초 대기 — 지연/hang/예외 발생 여부
  5. 확인 후 프로세스 종료 (`kill <PID>`) 및 잔존 프로세스 0개 확인
- 프론트엔드 수정 시: `npm run build` + 브라우저 화면 확인
- 아키텍처 원칙(이벤트 루프, 비동기 큐, WebSocket 등) 관련 수정은 런타임 기동 검증 생략 금지

프론트엔드 기본 검증: `npm run build`
빌드 성공만으로 브라우저 표시가 100% 보장되는 것은 아님. UI 문제는 실제 화면 확인 필요

### 임시방편 금지 목록
- !important 남발
- 타입 오류를 무조건 as any로 덮기
- 전역 CSS에 페이지 전용 스타일 추가
- service worker 캐시 문제를 무시하고 코드만 계속 수정
- 사용자 승인 없이 범위 밖 리팩토링
- UI 디자인을 임의 변경

### 완료 보고 형식
- 수정한 파일
- 해결한 근본 원인
- 검증한 항목
- 남은 확인 사항
- 사용자가 직접 확인할 방법
