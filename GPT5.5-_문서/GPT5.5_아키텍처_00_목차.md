# Cascade 아키텍처 00 - 목차

## 목적

이 문서 세트는 `SectorFlow`를 일반 웹앱이 아니라 **한국 주식시장 1인 로컬 자동매매 HTS급 실시간 시스템**으로 보기 위한 Cascade 기준 아키텍처 참고서다.

목표는 다음이다.

- 하위 AI가 프로젝트 성격을 오해하지 않도록 한다.
- 쇼핑몰/홈페이지식 CRUD 아키텍처를 배제한다.
- 실시간 데이터 지연, 블로킹, 왜곡 가능성을 최우선 제거 대상으로 본다.
- 현재 코드의 위험 지점을 정확히 식별한다.
- 궁극적으로 도달해야 할 최적 아키텍처와 단계별 로드맵을 제시한다.

## 하위 AI가 반드시 지켜야 할 전제

이 프로젝트는 다음이 아니다.

```text
쇼핑몰
블로그
홈페이지
일반 관리자 페이지
느슨한 CRUD 웹앱
```

이 프로젝트는 다음이다.

```text
한국 주식시장 실시간 데이터 기반
1인 로컬 자동매매 시스템
HTS급 이상의 반응성과 안정성을 지향하는 앱
실시간 데이터 무결성과 지연 최소화가 핵심인 시스템
```

## 최우선 판단 기준

모든 설계/수정/리팩토링은 다음 질문을 통과해야 한다.

```text
이 변경이 실시간 데이터 수신을 지연시키는가?
이 변경이 이벤트 순서를 왜곡하는가?
이 변경이 hot path에 로그/I/O/무거운 렌더링을 추가하는가?
이 변경이 주문/체결 상태를 불명확하게 만드는가?
이 변경이 프론트 메인 스레드를 막는가?
이 변경이 캐시/상태/DI 인스턴스를 분산시키는가?
```

하나라도 그렇다면 HTS급 아키텍처에 부합하지 않는다.

## 최종 목표 한 줄 요약

`SectorFlow`의 궁극적 목표는 **로컬 단일 사용자 환경에서 브로커 실시간 이벤트를 최소 지연으로 수신하고, 데이터 왜곡 없이 전략/주문/체결/포지션/UI까지 전달하는 HTS급 자동매매 아키텍처**다.
