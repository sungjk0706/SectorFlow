# Cascade 문제해결 00 - 목차

## 목적

이 연재 문서는 `SectorFlow` 프로젝트에서 발생하는 다양한 문제를 하위 AI가 반복 삽질 없이 조사, 분석, 해결, 검증할 수 있도록 만든 포괄적 참고서다.

핵심 목표는 다음과 같다.

- 문제를 추측이 아니라 코드와 로그로 확인한다.
- 증상 완화가 아니라 근본 원인을 제거한다.
- 수정 전 조사, 수정 후 검증을 분리한다.
- 사용자가 보고만 요청한 경우 승인 없이 수정하지 않는다.
- 토큰 낭비를 줄이기 위해 조사 순서와 검증 기준을 표준화한다.

## 문제 유형별 먼저 볼 문서

### UI가 안 보이는 문제

먼저 볼 문서:

```text
Cascade_문제해결_02_조사순서
Cascade_문제해결_03_UI스크롤
```

확인 순서:

1. 컴포넌트가 실제 코드에 있는가
2. route/container에서 렌더링되는가
3. props/state가 전달되는가
4. CSS가 숨기고 있는가
5. 스크롤 체인이 끊겼는가
6. 캐시/서비스워커가 오래된 화면을 보여주는가

---

### 버튼/입력값이 동작하지 않는 문제

먼저 볼 문서:

```text
Cascade_문제해결_02_조사순서
Cascade_문제해결_01_원칙
```

확인 순서:

1. 이벤트 핸들러가 연결되어 있는가
2. 핸들러가 호출되는가
3. store/state가 변경되는가
4. 변경된 state가 다시 UI에 반영되는가
5. 타입 오류나 값 변환 문제가 있는가

---

### 빌드 오류 문제

먼저 볼 문서:

```text
01_근본해결_원칙
02_조사_분석_표준순서
```

확인 순서:

1. 오류 메시지를 그대로 읽는다
2. 첫 번째 실제 소스 오류를 찾는다
3. 타입 정의와 호출부를 함께 확인한다
4. 임시 `any` 남발보다 타입 경로를 이해한다
5. 수정 후 `npm run build`로 검증한다

---

### 화면이 예전 상태로 보이는 문제

먼저 볼 문서:

```text
03_UI_CSS_스크롤_문제해결
```

확인 순서:

1. 개발 서버가 최신 번들을 제공하는가
2. 브라우저 강력 새로고침을 했는가
3. 서비스워커가 등록되어 있는가
4. 캐시가 남아 있는가

## 하위 AI 행동 규칙 요약

- 사용자가 “보고하라”고 하면 수정하지 말고 보고만 한다.
- 사용자가 “진행하라”고 승인하면 작은 단위로 수정한다.
- 긴 자동 치환 명령보다 작은 패치를 선호한다.
- 검색은 복잡한 정규식보다 고정 문자열 검색을 우선한다.
- 수정 후에는 반드시 검색 검증과 빌드 검증을 한다.
- 확실하지 않은 내용은 “확인 필요”라고 표시한다.
