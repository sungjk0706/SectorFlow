## SectorFlow 데이터 왜곡 현상 정밀 조사 보고서

정밀 조사 결과, 좌측 업종순위 테이블의 **업종상승률 100% 왜곡**과 **매수후보 테이블의 비정상적 동작**은 모두 **단 하나의 근본 원인(백엔드의 등락률 파싱 버그)**에서 비롯된 것임을 확인했습니다. (이전에 제가 수행한 파이프라인 분리 작업이 원인이 아니며, 오히려 파이프라인이 정상화되면서 기존에 숨어있던 파싱 버그가 수면 위로 드러난 것입니다.)

### 1. 우측 테이블(정상)과 좌측 테이블(100% 왜곡)이 다르게 표시된 근본 원인

**문제의 진원지: `engine_account_rest.py`의 `_parse_float_loose` 함수**
키움증권에서 전달받은 실시간 틱 데이터(FID 12, 등락률)를 처리할 때, 백엔드 내부의 `_parse_float_loose` 함수가 사용됩니다. 
이 함수 코드를 보면 아래와 같이 작성되어 있습니다.
```python
def _parse_float_loose(v) -> float:
    try:
        # 치명적 버그: lstrip("-")가 음수 기호를 강제로 날려버림
        cleaned = str(v).replace(",", "").replace("%", "").lstrip("-")
        return float(cleaned or 0)
    except (ValueError, TypeError):
        return 0.0
```
- **백엔드의 왜곡 (좌측 테이블 100%의 이유)**: `lstrip("-")` 코드가 모든 음수 기호를 제거해버립니다. 따라서 현대오토에버가 `-4.05%` 하락해도 백엔드의 `master_stocks_cache`에는 양수인 `4.05`로 저장됩니다. 업종 점수를 계산하는 `compute_sector_scores` 함수는 이 데이터를 바탕으로 `change_rate > 0`인 종목 수를 세어 상승비율을 계산하므로, **하락한 종목도 모두 상승한 것으로 착각**하여 상승비율이 항상 100%로 도출된 것입니다.

- **프론트엔드는 정상 표시된 이유 (우측 테이블)**: 백엔드 파이프라인에서 키움증권의 원본 틱 데이터(음수 기호가 포함된 원시 데이터)를 `broadcast_queue`를 통해 프론트엔드로 그대로 전송합니다. 프론트엔드의 `hotStore.ts`는 자체 파싱 함수(`parseKiwoomNum`)를 가지고 있으며, 여기서는 `if (s.includes('-')) sign = -1;` 로직을 통해 음수를 정확히 처리하여 화면에 `-4.05`로 그려줍니다. 즉, **프론트엔드와 백엔드가 데이터를 다르게 해석**하고 있었습니다.

### 2. 매수후보 테이블 진단 및 의도 부합 여부

개발자님의 설계 의도: *"상위업종순위에 속한 종목들의 리스트들의 실시간데이터를 표시하고 매수설정 설정값에 만족할때 매수주문이 발동되는 것"*

**진단 결과**: 설계 의도 자체는 백엔드(`engine_sector_score.py`의 `build_buy_targets` 및 `_check_buy_target_reached`)에 정확히 반영되어 있습니다. 구조적 결함은 없습니다.

**비정상 동작의 진짜 원인**:
매수후보를 선정할 때 "상승률 제한(block_rise_pct)"과 "하락률 제한(block_fall_pct)" 매수 가드 필터(`apply_stock_guards`)를 거치게 됩니다.
그런데 위에서 언급한 **`lstrip("-")` 버그로 인해 백엔드가 판단하는 등락률이 무조건 양수**입니다.
- 종목이 `-7.5%` 폭락해도 백엔드는 `+7.5%` 폭등한 것으로 인식합니다.
- 결과적으로 "하락률 제한" 가드에는 영원히 걸리지 않으며, 엉뚱하게도 폭락한 종목이 "상승률 제한(예: 7% 이상 상승 시 차단)" 가드에 걸려 차단(`blocked_targets`)되는 현상이 발생합니다.
- 이로 인해 매수후보 순위 산정과 주문 발동 로직이 완전히 엉뚱한 데이터를 기반으로 돌아가고 있었기 때문에, 매수후보 테이블이 이상하게 동작했던 것입니다.

### 요약 및 결론
모든 문제는 `_parse_float_loose` 함수에 존재하는 `lstrip("-")` 단 9글자 때문입니다. 이전에는 틱 처리기 내부에 임시로 등락률을 중복 덮어씌우는 로직이 있어 이 파싱 버그가 우연히 가려져 있었으나, 파이프라인을 정석대로 정리하면서 진성 버그가 드러난 것입니다. 

(요청하신 대로 코드는 수정하지 않고 원인 보고만 마칩니다. 확인 후 수정 지시를 내려주시면 즉시 반영하겠습니다.)
