## 1. 로그 레벨 재정의 기준

| 레벨 | 기준 | 예시 |
|------|------|------|
| **정보(INFO)** | 꼭 필요한 운영 로그만 | 앱 시작/종료, 장개시/장마감, 에러/경고 발생, 구독 등록/해지 완료(1회성) |
| **디버그(DEBUG)** | 개발/디버깅용 상세 로그 | 실시간 데이터 처리 흐름, 청크별 응답, 1초 통계, 화면전송 성공 상세 |
| **경고(WARNING)** | 주의가 필요한 상황 | 지연 50ms 초과, 구독 응답 시간 초과, 저장데이터 불완전, 토큰 발급 실패 |
| **오류(ERROR)** | 장애 발생 시 | 지연 200ms 초과(자동매매 중단), 처리 예외, 연결 실패 |

## 2. 단계별 수정 대상

### 1단계: 실시간 데이터 흐름의 불필요한 로그 제거 또는 디버그 레벨로 변경

**대상** (UI 기준 용어):
- **"실시간 데이터 수신 부분"**: 1초마다 `WS REAL 수신 개수` 로그 → **코드에서 완전 삭제**
- **"화면전송 성능 측정 부분"**: 1초마다 `브로드캐스트 시도/실제/비율` 로그 → **코드에서 완전 삭제**
- **"체결 처리 지연 측정 부분"**: 1초마다 `틱 개수/평균지연/최대지연` 로그 → **코드에서 완전 삭제**
- **"거래대금 FID14 상세 부분"**: 10초마다 찍히는 거래대금 상세 → **코드에서 완전 삭제**
- **"WS 응답 상세 부분"**: REG/UNREG 응답 상세, REAL 항목 미리보기 → 디버그 레벨 하향 (이미 디버그지만 확인)

**예상 수정 범위**: 3~4군데
**영향 기능 영역**: 실시간 데이터 수신 → 화면전송 흐름 전체

**검증 방법**:
1. `python3 -m py_compile` 컴파일 확인
2. `grep -r "WS REAL 수신\|브로드캐스트 시도\|1초 틱\|FID14.*거래대금" backend/` → **결과 없음** (코드에 해당 문자열 없음)
3. 해당 통계 누적/플러시 함수(`_record_real_received`, `_flush_broadcast_stats`, `_flush_tick_stats`, `_update_trade_amount_fid14` 및 관련 전역 변수)도 함께 제거되었는지 확인

### 3단계: 로그 포맷 통일

**접두어 통일 기준**:

| 분류 | 통일 접두어 | 적용 대상 |
|------|------------|----------|
| 앱 시작/종료 관련 | `[시작]` | 앱 기동, 부트스트랩, 데이터준비 |
| 실시간 연결 관련 | `[연결]` | WS 접속/해제, 실시간 구독 상태 |
| 데이터 수신/전송 관련 | `[데이터]` | 체결가, 호가, 지수 수신 및 브로드캐스트 |
| 구독 관리 관련 | `[구독]` | 종목/지수/계좌 구독 등록/해지 |
| 타이머/스케줄러 관련 | `[타이머]` | 장개시/장마감, 확정데이터 조회 예약 |
| 성능 측정 관련 | `[성능]` | 1초 통계, 지연 측정 (DEBUG 레벨) |
| 에러/경고 | `[오류]`, `[경고]` | 예외, 지연 초과, 실패 |

**기존 → 신규 접두어 매핑**:
- `[엔진]` → `[데이터]` 또는 기능에 맞게 변경
- `[앱준비]` → `[시작]`
- `[실시간연결]` → `[연결]`
- `[구독등록]` → `[구독]`
- `[구독해지]` → `[구독]`
- `[통계]` → DEBUG 레벨로 내리면서 접두어 `[성능]`으로 변경
- `[거래대금]` → `[데이터]`
- `[체결강도]` → `[데이터]`
- `[지수]` → `[데이터]`
- `[모니터링]` → `[데이터]`
- `[파이프라인]` → `[타이머]`
- `[데이터동기화중]` → `[타이머]`

**기타 통일 항목**:
- **로그 메시지 형식 통일**: `완료 --`, `실패:`, `생략` 등 표현 통일
- **불필요한 이모지/특수문자 제거**: `⚫` 등 시각적 기호 제거

**불변 원칙**:
- 로그 포맷 변경 시 기존 로그 파싱에 의존하는 코드가 없음을 확인 완료
- 프로젝트 코드(`.venv` 제외)에서 로그 파일을 `open/read`로 읽어 파싱하는 코드 없음
- 외부 라이브러리(loguru, uvicorn 등)에서도 로그 문자열 기반 파싱 없음

**예상 수정 범위**: 전체 로그 출력 라인 대상 (35개 파일, 500+ 라인)
**영향 기능 영역**: 전체 백엔드

**검증 방법**:
1. `python3 -m py_compile` 컴파일 확인
2. 로그 파일 10줄을 무작위로 샘플링해서 접두어와 형식이 통일되었는지 확인
3. 기존 접두어(`[엔진]`, `[앱준비]` 등)가 남아있지 않은지 확인

- **"타이머 콜백 오류 부분"**: 파이프라인 단계별 실패 원인 구체화

**실제 수정 범위**: 5개 파일, 16건
- `ws_subscribe_control.py`: 계좌 구독 보장/지수 구독/실시간시세 구독/자동 구독 ×2 — `exc_info=True` 추가 (5건)
- `trading.py`: 텔레그램 알림/일일 매수 복원/보유 종목 조회/테스트모드 UI 갱신 ×2 — `exc_info=True` 추가 (5건)

- `daily_time_scheduler.py`: 시간 파싱/설정 로드 ×2 — 에러 삼킴 → `logger.warning(..., exc_info=True)` 추가 (3건)

**검증 결과**:

## 3. 예상 영향 및 주의사항

- **실시간 무결성**: 로그 레벨/접두어 변경만이므로 데이터 처리 로직에 영향 없음

- **디버깅 가이드**: 향후 실시간 흐름 디버깅 시 `trading_debug.log`에 DEBUG 레벨이 기록되지 않음 (현재 INFO 이상으로 제한 중). 필요 시 `logger.py`의 `_debug_file_sink` 레벨을 DEBUG로 복원해야 함
- **금지 패턴 위반 여부**: 없음 (로그 레벨/문자열만 변경, 락/폴링/큐 관련 코드 미투입)
