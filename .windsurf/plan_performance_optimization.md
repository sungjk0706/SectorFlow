# 주식 자동매매 앱 성능 최적화 단계별 계획서 (백프레셔 제외, 현 상태 반영)

## 현재 구현 상황 (사용자 제공)
- 웹소켓 실시간 연결: 구현 중
- 프론트엔드 렌더링: DOM 증분 방식 + 가상 스크롤링 + Canvas 차트 적용 중
- 시계열 DB: 장중에는 불필요하므로 제외 (히스토리 조회 시 필요하면 별도 논의)

## 최종 목표
HTS 수준의 지연 없는 실시간 데이터 처리 (초당 100~500 틱 이상 안정적 처리, 60fps 유지)

## 단계별 계획

### 단계 1: 압축 로직(Conflation) 도입
**심층 조사**
- 현실 주식 데이터에서 동일 가격 반복 틱이 발생하는 패턴 분석 (초당 수백 건 가능)
- 최적의 시간 임계값(ms) 및 가격 변화 임계값 조사 (권장: 50ms, 1원 또는 종목별 호가 단위)
- 다양한 압축 전략: 시간 기반(일정 시간 내 마지막 틱만), 가격 변화 기반(일정 금액 이상 변할 때), 혼합

**분석**
- 현재 웹소켓 수신 데이터를 그대로 UI에 전달하면 불필요한 렌더링 발생
- CPU 사용률 및 배터리 소모 증가, 차트 갱신 과부하

**수정방향**
- 웹소켓 메시지 핸들러에 압축 필터 추가 (마지막 전송 틱과 동일 가격 & 50ms 내면 버림)
- 압축 조건 설정값을 환경변수 또는 설정 파일로 관리

**검증**
- 더미 데이터로 1초에 500개 틱 입력 후 실제 UI 갱신 횟수 측정 (압축 전: 500회 → 후: 약 20~50회)
- Chrome Performance 탭에서 CPU 사용량 비교 (압축 전/후)

---

### 단계 2: 논블로킹 I/O 전면 적용
**심층 조사**
- 현재 백엔드에서 사용 중인 DB 드라이버, 파일 I/O, 네트워크 호출이 블로킹인지 확인
- async/await 패턴과 논블로킹 라이브러리 (Node.js: fs.promises, net, http2; Python: asyncio, aiofiles)
- 이벤트 루프 블로킹이 실시간 데이터 처리에 미치는 영향

**분석**
- 블로킹 I/O 발생 시 메인 이벤트 루프가 멈춰 웹소켓 데이터 처리 지연 발생
- 특히 주문 전송, 거래 로그 저장, 설정 파일 읽기 등에서 병목 가능성

**수정방향**
- 모든 I/O 작업을 비동기 함수로 전환 (Promise, async/await)
- 동기식 함수(fs.readFileSync, execSync, time.sleep 등) 제거
- 데이터베이스 쿼리도 비동기 드라이버 사용 (예: better-sqlite3 대신 sqlite3 async)

**검증**
- 로그에 각 I/O 작업 시작/종료 시간 기록, 블로킹 여부 확인
- 스트레스 테스트 중 이벤트 루프 지연 시간 측정 (Node.js: perf_hooks.performance, Python: loop.slow_callback_duration)

---

### 단계 3: 인메모리 캐시 구축
**심층 조사**
- 자주 조회되는 데이터: 현재가, 호가, 잔고, 포지션, 실시간 지표 값
- 로컬 환경에서 Redis vs 메모리 변수(Map, Object) 성능 비교 (변수 캐시로 충분, Redis는 과할 수 있음)
- 캐시 갱신 전략 (TTL, 이벤트 기반, 실시간 웹소켓 수신 시 즉시 갱신)

**분석**
- 현재 매 틱마다 DB에 현재가 저장하거나 조회하면 병목 심함
- 실시간 매매 결정에 필요한 데이터는 밀리초 단위 접근 필요

**수정방향**
- 백엔드에 전역 캐시 모듈 생성 (예: Node.js의 Map, Python의 dict)
- 웹소켓으로 수신된 데이터는 먼저 캐시에 업데이트, 필요한 경우(일정 주기 또는 종료 시)에만 DB에 비동기 저장
- 프론트엔드 요청(현재가, 잔고 등) 시 캐시에서 즉시 응답

**검증**
- 캐시 미사용 vs 사용 시 응답 속도 측정 (예: 현재가 조회 API 응답 시간)
- 메모리 사용량 측정 및 캐시 크기 제한 (LRU 알고리즘 적용 가능)

---

### 단계 4: 전체 통합 및 성능 검증
**심층 조사**
- 엔드투엔드 레이턴시 측정 방법 (웹소켓 수신 시각 → UI 렌더링 완료 시각)
- 장시간 안정성 테스트 (8시간 연속 실행, 메모리 누수 확인)
- 병목 지점 재발견을 위한 프로파일링 도구 사용법

**분석**
- 각 단계별 최적화가 실제 사용자 경험에 미치는 영향 평가
- 압축 로직 임계값 튜닝 필요성 검토

**수정방향**
- 통합 모니터링 로그 추가 (초당 처리 틱 수, 평균 레이턴시, 프레임 드롭 수)
- 예외 상황(네트워크 끊김, 증권사 API 오류) 처리 보강
- 필요시 추가 최적화 (예: 메모리 풀, 오브젝트 재사용)

**검증**
- 실제 증권사 모의투자 API 연동 후 1시간 이상 운영
- Chrome DevTools Performance 탭에서 60fps 유지 확인
- 8시간 후 메모리 증가율 10% 미만 확인 (Chrome Memory 탭, Node.js 힙 스냅샷)

## 진행 프로세스 (단계별 공통)

각 단계는 아래 순서로 진행한다. 사용자 승인 없이 코드 수정 금지.

```
선조사(코드/파일 읽기) → 후분석(문제 원인 도출) → 수정계획 보고 → 사용자 승인 → 수정계획 진행 → 검증 → HANDOVER.md 업데이트 → 다음 단계
```

### 선조사
- 해당 단계와 관련된 모든 소스 파일을 읽어 현재 구현 상태를 파악
- 추측 금지. 파일 읽고 확인한 사실만 기술

### 후분석
- 선조사 결과를 바탕으로 병목 원인, 불필요한 작업, 개선 가능 지점 도출
- 영향 범위(수정 대상 파일, 연동 파일, 동작 변경 여부) 명시

### 수정계획 보고
- 구체적인 수정 내용(추가/변경/제거 대상)과 예상 효과를 보고
- **함수명 사용 자제**. UI 기준 용어(예: "실시간 시세 화면", "매수후보 목록", "계좌 탭")로 대체
- 일반 용어로 보고. 기술적 내부 구현명보다 사용자 경험 중심 표현 사용
- 보고 형식: `[확인한 사실]` / `[승인 요청]` 변경내용/영향범위

### 검증
- 수정 완료 후 반드시 테스트 수행, 결과 포함 보고
- `python3 -m py_compile` 또는 `npx tsc --noEmit` 등 자동 검증
- Chrome Performance/Memory 탭, 로그 출력 등 실측 결과 포함

### HANDOVER.md 업데이트
- 매 단계 완료마다 `.windsurf/HANDOVER.md`에 해당 Phase 기록 추가
- 완료 시점, 수정 파일, 수정 지점, 검증 결과, 기존 동작 변경 여역 명시

## AI 에이전트에 대한 지시사항 (반드시 지킬 것)

1. 위 단계별 계획서를 기준으로, **현재 구현 상태를 먼저 분석**해줘.
2. 각 단계에서 **심층 조사** 내용을 바탕으로 구체적인 코드 수정안을 제시해줘.
3. 각 단계 완료 후 **검증 방법**에 따른 테스트 결과를 보고해줘.
4. **백프레셔는 구현하지 않음**. 압축 로직(Conflation)이 핵심 대체 기술임.
5. **시계열 DB는 장중 사용하지 않음** → 제외. 히스토리 데이터는 필요시 별도 논의.
6. **단계 1부터 순서대로 진행**해줘. 한 단계가 완료되고 검증되면 다음 단계로 넘어가.
7. **사용자 명시적 승인 없이 코드 수정 금지**. 분석/제안은 자유.

**시작**: 단계 1부터 진행. 먼저 현재 웹소켓 데이터 수신 코드를 읽고, 압축 로직을 추가하는 구체적인 코드 변경안을 제시해줘.
