# HTS급 주식자동매매 파이프라인 아키텍처 리팩토링

현재 거대 단일 파일(`engine_service.py`)로 집중된 병목 구조를 해소하고, 초당 수백 번의 틱 데이터를 지연 없이 처리할 수 있는 **초고성능 분산 파이프라인(Event-Driven Architecture)**으로 전면 리팩토링합니다.

## User Review Required
> [!WARNING]
> 이 리팩토링은 백엔드의 핵심 뼈대(Core Engine)를 통째로 뜯어고치는 대공사입니다. 기존에 잘 작동하던 `engine_service.py`의 기능들이 여러 개의 전용 배관(Pipeline)으로 쪼개집니다. 

## Proposed Architecture (파이프라인 분리 구조)

기존 시스템을 **5개의 독립된 전용 배관(Pipeline)**으로 분리하여 각자의 역할만 비동기로 완벽하게 수행하도록 설계합니다.

### 1. Ingestion Pipeline (시세 전용 고속 수신부)
- **역할:** 키움증권 웹소켓에서 쏟아지는 틱/호가 데이터를 받아서 가공 없이 즉시 `Tick Queue`에 밀어 넣습니다.
- **특징:** 연산이나 로직 판단이 1%도 섞이지 않은 완벽한 I/O 전용 배관입니다.

### 2. Compute/Strategy Engine (초고속 연산 및 전략 두뇌)
- **역할:** `Tick Queue`에서 데이터를 꺼내어 **업종 순위 계산**, **체결강도 갱신**, **매수/매도 타점 도달 여부**를 0.001초 단위로 계산합니다.
- **분배:** 계산 결과 타점 도달 시 `OMS Queue`로 '주문 지령'을 던지고, UI 업데이트가 필요하면 `Broadcast Queue`로 결과물을 던집니다.

### 3. OMS Pipeline (주문 및 체결 전용 배관)
- **역할:** 연산 엔진이 던진 지령(`OMS Queue`)을 주워 담아 키움증권 REST/WS API를 통해 **실제 매수/매도 주문 네트워크 통신**을 수행합니다.
- **특징:** 느린 네트워크 I/O가 발생해도 실시간 시세 수신(1번 배관)이나 랭킹 계산(2번 배관)에는 전혀 병목을 주지 않습니다.

### 4. Broadcaster/Gateway (UI 전송 계층)
- **역할:** `Broadcast Queue`에 쌓인 정제된 결과물(엑기스)을 모아서(`backend_coalescing.py` 활용) 프론트엔드 웹 UI로 부드럽게 쏴줍니다.

### 5. Control Plane (사용자 설정 우회 배관)
- **역할:** 사용자가 앱에서 "매수 정지"나 "목표가 변경" 등을 누를 경우, 수만 개의 틱 데이터 줄을 서지 않고 즉각 연산 엔진(두뇌)의 상태값을 업데이트하는 VIP 우회 통신망입니다.

## Verification Plan

### 단계별 테스트 전략
1. **분리된 수신부 테스트:** 키움 웹소켓 틱이 `Tick Queue`에 딜레이 없이 쌓이는지 로그 검증
2. **연산조 고립 테스트:** 무거운 파이썬 딕셔너리 연산을 분리한 후, 메모리 누수 및 CPU 스파이크가 튀지 않는지 확인
3. **주문/체결 파이프라인 비동기 테스트:** 주문을 넣는 동안 시세 수신부가 단 1ms라도 블로킹(멈춤)되는지 벤치마크 검증
4. **통합 스트레스 테스트:** 장 초반(09:00) 트래픽을 가정한 Mock Data를 쏟아부어, [수신 -> 연산 -> 주문 -> 화면 갱신] 흐름이 끊기지 않는지 확인
